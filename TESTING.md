# Политика тестирования

**Проект:** Интерактивная доска для распределения задач (Tokboard)
**Ответственный:** Кривоносов Д.И. (тестирование)

## 1. Цели

Обеспечить стабильную работу веб-сокет-соединений, корректное распределение задач в реальном времени и надёжное хранение данных в PostgreSQL.

## 2. Уровни тестирования

**2.1. Модульное тестирование**
- **Фронтенд (Vue + Pinia):** тестирование компонентов, хранилищ Pinia и функций работы с веб-сокетами (библиотека Vitest).
- **Бэкенд (Node.js + Express):** тестирование контроллеров, middleware и утилит (библиотека Jest).
- **Покрытие:** не менее 70% для критической логики(код, без которого не работает основной функционал проекта) т.е (аутентификация, обновление досок через сокеты).

**2.2. Тестирование веб-сокетов**
- Отдельный набор тестов на падение соединения, переподключение (reconnection) и целостность данных после разрыва.
- Имитация одновременных подключений (до 10 клиентов).

## 3. Инструменты

__Уровень__  __Инструмент__ 

 Unit (фронт) - Vitest + @vue/test-utils 
 Unit (бэк) -  Jest 
 Интеграционное - Supertest (API) + тестовый PostgreSQL 
 Покрытие кода - Istanbul (nyc) / Vitest --coverage 
 CI/CD - GitHub Actions 

## 4. Процесс в CI/CD (GitHub Actions)

При пуше в ветку `main` выполняются следующие шаги:

1. **Установка зависимостей** (`npm ci` для клиента и сервера).
2. **Запуск модульных тестов** (фронт + бэк). При падении — остановка.
3. **Поднятие тестовой PostgreSQL** (контейнер) и запуск интеграционных тестов.
4. **Запуск E2E тестов** (Playwright) в headless-режиме.
5. **Сборка отчёта о покрытии**. Падение, если общее покрытие < 70%.
6. **Деплой** происходит только при успешном прохождении всех тестов.

## 5. Тестовые данные

- Для интеграционных тестов: отдельная схема БД, которая пересоздаётся перед каждым прогоном (`prisma migrate reset --force`).
- Секреты (URL БД, JWT-ключ) передаются через GitHub Secrets.

## 6. Роли в тестировании
     
 Кривоносов Д.И. (тестирование) - Написание интеграционных тестов, настройка CI, управление багами, проверка качества перед релизом, написание unit-тестов для серверной логики, Vue-компонен и хранилищ Pinia, мокирование веб-сокетов в тестах, обеспечение чистоты тестовой БД.
 
 Архипов П.Н. (лидер) -  Утверждение политики, контроль метрик покрытия, финальное решение по качеству. 

## 7. Порядок при обнаружении бага 

1. Тестировщик (Кривоносов) создаёт issue в GitHub с меткой `bug` и описанием шагов. (опционально)
2. Разработчик ответственный за свою часть разработки пишет тест, который воспроизводит баг. (опционально)
3. Разработчик исправляет код.
4. CI прогоняет все тесты. Если тест проходит — баг закрывается.

## 8. Критерии готовности к релизу

- Все тесты (unit, интеграционные) проходят в CI.
- Покрытие кода 70%.
- Отсутствуют открытые критические баги
- Проведено ручное исследовательское тестирование веб-сокетов (имитация обрыва связи, быстрых переключений между досками).

## 9. Запуск тестов локально (для разработчика)
