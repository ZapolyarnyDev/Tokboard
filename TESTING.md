# Политика тестирования программного обеспечения

## 1. Область применения

Документ устанавливает обязательные требования к тестированию приложения Tokboard. Распространяется на разработчиков и тим.лида.

## 2. Термины

- **Модульное тестирование** — проверка отдельного модуля в изоляции от внешних зависимостей.
- **Интеграционное тестирование** — проверка взаимодействия двух и более модулей (БД, API, очереди).
- **Сквозное тестирование** — проверка полного пользовательского сценария в среде, приближенной к продуктивной.
- **Покрытие кода** — доля исполненных строк кода при прогоне тестов (в процентах).

## 3. Уровни тестирования

**3.1. Модульное тестирование**
- Исполнитель: Тестер
- Обязательность:тесты для всего нового кода.
- Покрытие: не менее 80% (бизнес-логика), не менее 60% (вспомогательный код).
- Запрещено: обращение к сети, файловой системе, БД. Зависимости заменяются моками.

**3.2. Интеграционное тестирование**
- Исполнитель: Тестер
- Обязательность: для всех модулей с внешними взаимодействиями.
- Покрытие: все успешные сценарии + минимум 2 негативных на интерфейс.

**3.3. Сквозное тестирование (E2E)**
- Исполнитель: Тестер
- Обязательность: для критических пользовательских путей.
- Запуск: при каждом мерже в целевую ветку.

## 4. Инструменты

По согласованию с тим.лидом

## 5. Процесс выполнения тестов

**5.1. Локально (до коммита)**
- `make test-unit` — обязателен.
- `make test-integration` — при изменении интеграционных слоёв.

**Правило:** падение любого теста останавливает конвейер. Мерж блокируется до исправления.

## 6. Требования к качеству тестов

**6.1. Flaky-тесты**
- Тест признаётся flaky при 2+ падениях за 5 прогонов без изменений кода.
- Исправление или удаление — в течение 48 часов.
- Временное решение: автоматический перезапуск (не более 2 раз) с созданием задачи.

**6.2. Производительность**
- Unit: 0,5 сек
- Интеграционный: 5 сек
- E2E: 30 сек

**6.3. Изоляция**
- Тесты не должны зависеть от порядка выполнения.
- Каждый тест обязан очищать свои артефакты (БД, файлы, кэш).

## 7. Тестовые данные

- Использование продуктивных данных в тестах запрещено.
- Разрешены: синтетические данные (Faker), анонимизированные копии (по разрешению ИБ).
- Секреты (логины, пароли, токены) не фиксировать в репозитории. Использовать переменные окружения или secrets CI/CD.

## 8. Порядок действий при дефекте


1. Написать тест, воспроизводящий дефект 
2. Зарегистрировать дефект в трекере с меткой «требуется тест» 
3. Устранить дефект  -По приоритету-
4. Запустить все тесты в CI/CD  Автоматически 
5. Для дефектов уровня Blocker — провести постмортем 

## 9. Ответственность

- **Нарушение покрытия:** коммит не принимается. Исключение - если разрешит ZapolyarnyDev
- **Отсутствие тестов для критического пути:** технический долг с приоритетом задачи.
